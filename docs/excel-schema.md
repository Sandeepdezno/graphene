# Graphene — Excel Workbook Contract (`flagship.xlsx`)

**Tier legend:** 🎬 Investor Demo · 🏗️ Production MVP · 🚀 Long-Term Platform

This is the contract the Excel importer (GRAPH-D1.2) parses. It is designed
together with the flagship demo workbook (`demo-data/flagship.xlsx`, generated by
`backend/scripts/generate_flagship_dataset.py`). A change to this contract is a
change to the parser and the generator both.

The workbook has **five sheets**. Each produces nodes and relationships in the
graph (`03-graph-schema.md`). Column headers are exact and case-sensitive.

---

## Sheet 1 — `Programs`

One row per program.

| Program | Description | BusinessMeaning | Reads | Writes | Calls |
|---|---|---|---|---|---|

- **Program** — the program's name (node identity). Label: `Program`.
- **Description**, **BusinessMeaning** — free text shown in the node drawer.
- **Reads** — comma-separated **Table** names → one `READS` edge each (Program→Table).
- **Writes** — comma-separated **Table** names → one `WRITES` edge each (Program→Table).
- **Calls** — comma-separated **FunctionModule** names → one `CALLS` edge each (Program→FunctionModule).

Reads / Writes / Calls cells may be blank.

## Sheet 2 — `FunctionModules`

One row per function module. Same columns and edge semantics as `Programs`, with
the row name labelled `FunctionModule`.

| FunctionModule | Description | BusinessMeaning | Reads | Writes | Calls |
|---|---|---|---|---|---|

## Sheet 3 — `BAdIs`

One row per BAdI implementation.

| BAdI | Description | BusinessMeaning | ImplementedBy |
|---|---|---|---|

- **BAdI** — the BAdI name. Label: `BAdI`.
- **ImplementedBy** — the **Program** that contains the implementation → one
  `IMPLEMENTS` edge, directed **Program → BAdI** (the program implements the BAdI).

## Sheet 4 — `Jobs`

One row per background job.

| Job | Description | BusinessMeaning | Executes |
|---|---|---|---|

- **Job** — the job name. Label: `Job`.
- **Executes** — comma-separated **Program** names → one `EXECUTES` edge each,
  directed **Job → Program**.

## Sheet 5 — `Transports`

One row per transport request. (Note: no `BusinessMeaning` column.)

| Transport | Description | Changed |
|---|---|---|

- **Transport** — the transport id. Label: `Transport`.
- **Changed** — comma-separated names of any existing objects → one `CHANGED`
  edge each, directed **Transport → object**.

---

## Contract rules

### 1. Node identity is the name

A node **is** its name. The same name appearing in multiple cells (or in a row and
a cell) refers to **one** node. Cell lists are comma-separated; surrounding
whitespace is trimmed.

### 2. Implicit nodes

Nodes may be introduced **implicitly** by appearing only inside a cell. The most
common case is a **Table**: there is no `Tables` sheet, so every Table exists only
because it appears in some `Reads`/`Writes` cell. Implicit nodes are created with
**empty** `description` and `business_meaning`. (A `Calls` target names a
`FunctionModule`; if it is not also rowed in the `FunctionModules` sheet it is
created implicitly the same way. `Changed` targets must name objects that already
exist elsewhere in the workbook, so their label is unambiguous.)

### 3. Provenance of direct relationships

Every relationship that comes from an explicit cell is stamped:

- `confidence = "direct"`
- `source = "excel_import"`
- `derived_from = "<sheet>!<cell-or-row-ref>"` — e.g. `Programs!C5` or
  `Programs!Z_PRICE_ENGINE.Writes`.

The same provenance model applies to nodes (`source = "excel_import"`,
`confidence = "direct"` for rowed nodes; implicit nodes carry `confidence =
"direct"` with an empty description).

### 4. Inference rule (the ONLY inference for the demo)

Keep it this simple — one rule, one hop:

> **If `A` CALLS `B` and `B` WRITES `T`, then `A` gets an inferred `WRITES` edge to `T`.**
> (Transitive write-through: what a program writes *through* a function module it calls.)

Inferred edges are stamped:

- `confidence = "inferred"`
- `source = "inferred"`
- `derived_from = "transitive: A→B→T"`

An inferred edge is **not** created if the same direct edge already exists.

**Worked example** (straight from the flagship workbook):

- `Programs` sheet: `Z_PRICE_ENGINE` **Calls** `Z_FM_VALIDATE_PRICE` → direct
  `(Z_PRICE_ENGINE)-[:CALLS]->(Z_FM_VALIDATE_PRICE)`.
- `FunctionModules` sheet: `Z_FM_VALIDATE_PRICE` **Writes** `ZTABLE_VALIDATION_ERRORS`
  → direct `(Z_FM_VALIDATE_PRICE)-[:WRITES]->(ZTABLE_VALIDATION_ERRORS)`.
- The rule fires:
  `(Z_PRICE_ENGINE)-[:WRITES {confidence:"inferred", source:"inferred", derived_from:"transitive: Z_PRICE_ENGINE→Z_FM_VALIDATE_PRICE→ZTABLE_VALIDATION_ERRORS"}]->(ZTABLE_VALIDATION_ERRORS)`.

This is the edge that powers the "we tell you what we *inferred*" beat of the demo:
`Z_PRICE_ENGINE` never names `ZTABLE_VALIDATION_ERRORS` in its own row, yet the graph
knows it writes there.

### 5. Edge summary

| Cell | Sheet | Edge | Direction |
|---|---|---|---|
| Reads | Programs / FunctionModules | `READS` | source → Table |
| Writes | Programs / FunctionModules | `WRITES` | source → Table |
| Calls | Programs / FunctionModules | `CALLS` | source → FunctionModule |
| ImplementedBy | BAdIs | `IMPLEMENTS` | Program → BAdI |
| Executes | Jobs | `EXECUTES` | Job → Program |
| Changed | Transports | `CHANGED` | Transport → object |

Relationship types `CREATED` and any node label `User` from `03-graph-schema.md`
are **out of scope** for this workbook contract.

### 6. Counting

- **Node count** = distinct names across all rows and all cells.
- **Direct edge count** = distinct `(source, type, target)` triples from explicit
  cells. Duplicate triples collapse to one edge.
- **Inferred edges are added on top by the importer** and are **not** part of the
  workbook's direct-edge total.
